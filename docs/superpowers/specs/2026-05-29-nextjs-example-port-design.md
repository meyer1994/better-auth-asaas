# Next.js Example Port — Design

## Goal

Build `examples/next/` as a feature- and layout-parity port of the current `examples/nuxt/` example. Same auth flow, same Asaas payment integration, same SQLite + Drizzle backend, same set of pages, same overall app structure — implemented with Next.js App Router idioms and shadcn/ui instead of Nuxt UI.

This supersedes `2026-05-26-nextjs-example-design.md`, which was written against an older version of the Nuxt example (separate `/charges/new` + `/charges/[id]` routes, server-component data fetching). The current Nuxt example consolidates everything into a single `/payments` page with client-side data fetching.

## Stack

- Next.js 15 (App Router) + React 19 + TypeScript
- shadcn/ui (already scaffolded — `components.json` present, Radix primitives + `clsx`/`tailwind-merge` in deps) + Tailwind v4
- Better Auth + `better-auth-asaas` (workspace dep)
- Drizzle ORM + `better-sqlite3`
- `@tanstack/react-query` for client-side data fetching (the React analogue of Nuxt's `useAsyncData`)
- `react-hook-form` + `zod` resolver for forms
- `sonner` for toasts

## Workspace Integration

- `examples/next` is already in `pnpm-workspace.yaml` and depends on `better-auth-asaas: workspace:*`.
- No new root scripts needed; existing `dev`/`build`/`typecheck` patterns apply.

## Pages In Detail

### `/` — Entry
Server component. Calls `auth.api.getSession({ headers: await headers() })`. If session exists → `redirect('/payments')`; otherwise → `redirect('/login')`. Mirrors Nuxt's `index.vue` intent without a client-side flash.

### `/login`
Client form. `react-hook-form` + Zod (email valid, password ≥ 8). On submit calls `authClient.signIn.email(...)`. Success → `router.push('/payments')`. Error → `sonner` toast.

### `/register`
Client form. Fields: `name`, `cpfCnpj`, `email`, `password`. Calls `authClient.signUp.email({ name, email, password, cpfCnpj })`; the `cpfCnpj` field flows through better-auth's `additionalFields` config. Success → redirect to `/payments`. The Asaas customer is auto-provisioned server-side by the plugin during sign-up.

### `/payments`
Client page, two-column grid (stacks on mobile):
- **Left column:**
  - Create-payment card: `react-hook-form` + Zod, fields `value` (number), `dueDate` (date), `billingType` (select: PIX / BOLETO / CREDIT_CARD), `description` (text). Submits via `useCreatePayment` mutation. Success toast + automatic table refresh via query invalidation.
  - Session debug card: pretty-printed `useSession()` output (matches the Nuxt example's debug card).
- **Right column:**
  - Payments table (columns: ID, Value, Due Date, Billing Type, Description, Actions). Fed by `usePayments()`. Manual "Refresh" button calls `refetch()`. Auto-refetches on window focus (TanStack default).
  - Row action button opens a shadcn `<Dialog>` showing the QR code image + a "Copy PIX payload" button. QR is fetched inside the modal via a scoped `useQuery({ queryKey: ['payment-qr', id], queryFn: () => authClient.asaas.payments.qr(id), enabled: open })`, wrapped in `<AsyncData>`.

### Route protection
Lightweight, same posture as the Nuxt example: no Next middleware. `/payments` reads `useSession()`; if it resolves to `null`, the page calls `router.replace('/login')`. The header and `/` redirect handle the rest.

### Header
Server component. Calls `auth.api.getSession`. Unauthenticated → "Login" + "Register" links. Authenticated → "Payments" link + `<LogoutButton />` client child that calls `authClient.signOut()` and refreshes.

## Data Layer

**`Providers`** (`"use client"`) mounts a single `QueryClient` (default `staleTime: 30s`, `retry: 1`, no retry on 401) plus shadcn's `<Toaster />`.

**`use-session.ts`** — thin wrapper around `authClient.useSession()`. We do not put session in TanStack Query — better-auth's hook is already reactive to sign-in/out, and mixing the two creates cache-invalidation churn around auth transitions. Same shape as Nuxt's `useSession()`.

**`use-payments.ts`** — `useQuery({ queryKey: ['payments'], queryFn: () => authClient.asaas.payments.list() })`. Returns the same `Page<Payment>` shape as the Nuxt composable.

**`use-create-payment.ts`** — `useMutation` calling `authClient.asaas.payments.create(input)`; on success runs `queryClient.invalidateQueries({ queryKey: ['payments'] })`. Toasts on success/error.

**QR fetch** — per-row scoped query, only enabled while the dialog is open (`enabled: open`), keyed by payment id so multiple rows cache independently.

**`<AsyncData>`** — render-prop component that takes any TanStack `UseQueryResult` and renders one of `loading | error | data` children. Direct port of the Vue component's role.

**Mapping to `useAsyncData`:** TanStack's `{ data, isLoading, error, refetch }` is the React analogue of Nuxt's `{ data, pending, error, refresh }`. Refetch-on-focus replaces Nuxt's implicit data refresh on navigation.

## Auth Configuration (`src/lib/auth.ts`)

Direct port of `examples/nuxt/server/utils/auth.ts`:

```ts
betterAuth({
  secret: process.env.BETTER_AUTH_SECRET!,
  baseURL: process.env.BETTER_AUTH_URL ?? 'http://localhost:3000',
  database: drizzleAdapter(db, { provider: 'sqlite', schema, debugLogs: true }),
  emailAndPassword: { enabled: true },
  user: { additionalFields: { cpfCnpj: { type: 'string', required: true, input: true } } },
  plugins: [
    asaas({
      apiKey: process.env.ASAAS_API_KEY!,
      sandbox: process.env.ASAAS_SANDBOX !== 'false',
    }),
  ],
})
```

**`src/lib/auth-client.ts`**: `createAuthClient({ plugins: [asaasClient()] })`. Mirrors `useAuth()` in the Nuxt composable.

**Auth route handler** (`src/app/api/auth/[...all]/route.ts`): `export const { GET, POST } = toNextJsHandler(auth)`.

## Webhook

The `asaas` plugin mounts `/api/auth/asaas/webhook` automatically through the catch-all handler — no extra route file. `PAYMENT_RECEIVED` / `PAYMENT_OVERDUE` / `PAYMENT_DELETED` are logged from the plugin config in `src/lib/auth.ts`, identical to the Nuxt example. Local tunnel + Asaas dashboard setup is documented in the README.

## Explicitly Dropped From Nuxt

- **`g h` / `g n` keyboard shortcuts** — Nuxt UI-specific affordance; no clean shadcn equivalent without a custom global key handler. Easy to add later via a `useEffect` if requested.

## Out Of Scope (YAGNI)

- Tests (Nuxt example has none either; verification is manual).
- Deploy / Docker config beyond `.env.example`.
- Theme switcher, i18n, dark mode toggle.
- Middleware-based route protection (matches Nuxt's lighter posture).
- Server actions / RSC data fetching for payments (explicitly chose client-side + TanStack Query to mirror Nuxt).

## Verification (manual, mirrors Nuxt)

1. `pnpm build` (workspace root) — builds the plugin.
2. `cd examples/next && cp .env.example .env`, fill in sandbox values.
3. `pnpm db:push` — creates SQLite schema.
4. `pnpm dev` — server at `http://localhost:3000`.
5. Sign up → confirm Asaas customer appears in sandbox dashboard.
6. Create a PIX charge → row appears in the table.
7. Open QR modal → QR image + PIX payload render; copy button works.
8. Expose dev server via `ngrok`, configure webhook in Asaas → trigger sandbox payment → check server logs for `PAYMENT_RECEIVED`.
